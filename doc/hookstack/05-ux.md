# 05 — UX d'implémentation des hooks sélectionnés

## Problème

Une fois les hooks sélectionnés sur Hookstack, l'utilisateur doit les implémenter dans son projet local. Le gap est **site web → fichiers locaux** :

1. Merger le fragment JSON avec son `.claude/settings.json` existant
2. Créer les fichiers scripts dans `.claude/hooks/`
3. Rendre les scripts exécutables si nécessaire

L'UX actuelle (copier-coller manuel du JSON + créer les scripts à la main) crée une friction inutile.

---

## Options analysées

| Solution | Setup requis | UX | Merge intelligent | Coût maintenabilité |
|---|---|---|---|---|
| **Script node généré** | Zéro | Coller dans terminal | Partiel (déterministe) | Très faible |
| **Prompt Claude Code généré** | Zéro | Coller dans Claude Code | Oui (IA) | Très faible |
| **Skill `implement-hooks`** | Une fois | `/implement-hooks` | Oui (IA) | Moyen |
| **NPX CLI** | `npx hookstack apply` | Terminal | Partiel | Élevé (npm publish) |
| **Plugin MCP** | Lourd (config MCP) | Automatique | Oui | Très élevé |

---

## Recommandation retenue : script node généré (path principal)

### Pourquoi ce choix

**Zéro friction, zéro dépendance.** L'utilisateur copie un script `.mjs` autogénéré, le colle dans son terminal, c'est fait. Pas besoin que Claude Code soit actif, pas de skill à installer, pas de npm.

La logique de merge (`mergeSettings()`) existe déjà côté client dans `mergeConfig.ts` — il suffit de la sérialiser dans le script généré. Coût d'implémentation : ~50 lignes.

**Le script généré :**
- Est spécifique à la sélection courante (JSON hardcodé)
- Lit le `.claude/settings.json` existant s'il existe
- Merge intelligemment (regroupe par event + matcher, pas d'écrasement)
- Crée les fichiers scripts dans `.claude/hooks/`
- Affiche un résumé clair de ce qui a été fait

### Ce que le script gère

```
✓ settings.json existant trouvé, fusion en cours...
✓ .claude/settings.json mis à jour
✓ .claude/hooks/detect-secrets.sh créé
✓ .claude/hooks/block-destructive.sh créé

✅ Installation terminée !
Relance Claude Code pour activer les nouveaux hooks.
```

### UX dans Hookstack

Bouton "Script d'installation" dans `HookConfigurator`, à côté de "Copier" et "settings.json". Un clic copie le script complet dans le presse-papier. L'utilisateur n'a qu'à faire `node install-hooks.mjs` dans son projet.

---

## Path secondaire : prompt Claude Code généré

Pour les cas où l'utilisateur a une config complexe ou des conflits à résoudre, générer un prompt riche à coller dans Claude Code :

> "J'ai sélectionné ces hooks sur Hookstack. Voici le fragment settings.json à merger : [...]. Voici les scripts à créer : [...]. Implémente-les dans mon projet en mergant avec mon .claude/settings.json existant."

L'IA gère les edge cases (conflits de matchers, chemins custom). Zéro setup, mais nécessite Claude Code actif.

**À implémenter dans une itération suivante.**

---

## Enhancement futur : skill `implement-hooks`

Pour les utilisateurs récurrents de Hookstack, une skill installable une fois qui donne accès à `/implement-hooks [url-de-sélection]`.

Problème de bootstrap : la skill doit être installée avant utilisation. Solution : un one-liner sur le site (`curl -s hookstack.dev/install-skill.sh | bash`) qui copie le `SKILL.md` dans `~/.claude/skills/`.

**À considérer quand Hookstack a une audience établie.**
