# Quand lire

Au **démarrage de toute session** et avant toute décision d'architecture, marketing
ou produit. Ce bundle est la source de vérité — prioritaire sur les suppositions.

# Comment lire (progressive disclosure)

1. Ouvrir [/index.md](/index.md) — la table des matières.
2. Ouvrir seulement les concepts pertinents à la tâche (ne pas tout charger).
3. Suivre les liens type `[texte](/vision/mission.md)` pour le contexte voisin.
4. Tolérer les liens cassés : ils marquent du savoir pas encore écrit, pas une erreur.

# Règles de consommation

- **Ne jamais supposer** ce que le bundle peut répondre — vérifier d'abord.
- Le bundle reflète l'état au moment de l'écriture (`timestamp`). Si un concept
  cite un fichier/flag/fonction, **vérifier qu'il existe encore** dans le code.
- En cas de contradiction code ⇄ bundle : le code fait foi, et **corriger le bundle**
  (voir [self-improvement](self-improvement.md)).

# Après la tâche

Si tu as appris quelque chose de non-trivial et durable (décision d'archi, contrainte,
fait marketing, étape de roadmap franchie), enrichis le bundle avant de clore.
Voir [self-improvement](self-improvement.md).
